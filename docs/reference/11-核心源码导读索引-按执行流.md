# 核心源码导读索引（按执行流）

> 基于 Claude Code v2.1.88 反编译源码分析  
> 目标：不是按目录罗列文件，而是按“一个请求从进入到完成”去读源码

## 0. 最推荐的阅读方式

不要先从 `tools/` 或 `screens/` 开始扫目录。

最有效的方式是顺着一条请求链读：

1. 用户输入怎么进入系统
2. 哪一层把它变成会话消息
3. 哪一层把它送进模型
4. 哪一层决定是否执行工具
5. 工具结果怎么回写
6. 系统怎么决定结束本轮还是继续下一轮

这一篇就是按这个顺序组织的。

## 1. 第一个入口：CLI 怎么把请求送进 runtime

### 文件

- `src/entrypoints/cli.tsx`
- `src/main.tsx`

### 先看什么

先看：

- fast-path 分流
- 主命令定义
- `--print` 模式入口

### 关键问题

读这层时重点只回答两个问题：

1. 哪些模式在最早阶段就被分流了？
2. 哪些模式最后仍然走同一条核心链路？

### 你会得到的结论

- CLI 负责“怎么启动”
- 但不负责“怎么跑一轮 agent”

这是后面所有设计成立的前提。

## 2. 第二个入口：headless/SDK 为什么没有复制一套逻辑

### 文件

- `src/cli/print.ts`
- `src/cli/structuredIO.ts`

### 先看什么

建议先看：

- `src/cli/print.ts:2170-2215`
- `src/cli/print.ts:4149-4323`
- `src/cli/structuredIO.ts:533-634`

### 重点理解

这里最关键的不是输出格式，而是：

- print 层如何把依赖喂给 `ask(...)`
- `StructuredIO` 如何把权限请求、hook callback、elicitation 变成控制协议
- 为什么 SDK 权限弹窗和本地 hook 可以 race

### 应该形成的认识

- `--print` 不是“精简版”
- 它是“同一个 runtime 的另一种壳”

如果你读完这层仍然觉得 print 是旁路，说明你还没抓到这个项目的主轴。

## 3. 会话协调层：为什么 `QueryEngine` 很重要

### 文件

- `src/QueryEngine.ts`

### 先看什么

建议顺序：

1. `class QueryEngine`
2. `submitMessage()`
3. `ask()`

对应位置：

- `src/QueryEngine.ts:184-209`
- `src/QueryEngine.ts:209-620`
- `src/QueryEngine.ts:1186-1288`

### 读这层时要带着的 5 个问题

1. 哪些状态跟着整个会话走，而不是只跟一轮 turn 走？
2. 用户输入在什么时候真正写入 transcript？
3. slash command、附件、memory prompt 是在哪一层接入的？
4. 为什么 `ask()` 可以复用全部主链路？
5. 为什么 system init message 是在这里发，而不是在 query loop 里发？

### 这层的核心角色

它是：

- session coordinator
- state owner
- transcript gateway
- SDK message normalizer

它不是：

- 模型采样器
- 工具调度器
- UI renderer

这一点分清楚后，后面的文件会顺很多。

## 4. 真正的主脑：`query.ts` 里的 turn machine

### 文件

- `src/query.ts`

### 先看什么

建议按下面顺序读：

1. `query()` 的壳
2. `queryLoop()` 开头的 immutable params / mutable state
3. 预处理与 compact 逻辑
4. API streaming 主循环
5. tool execution 汇合点
6. 递归进入下一轮的位置

重点位置：

- `src/query.ts:230-305`
- `src/query.ts:296-430`
- `src/query.ts:554-864`
- `src/query.ts:1011-1186`
- `src/query.ts:1366-1564`

### 读法

这一层不要逐行看，先抓 3 个问题：

1. 一轮 turn 的开始和结束边界在哪里？
2. 哪些逻辑发生在“调用模型之前”？
3. 哪些信号会导致继续下一轮？

### 你最终应该能复述出这条链

```text
prepare messages
  -> prefetch memory / skills
  -> compact / budget controls
  -> call API streaming
  -> assemble assistant blocks
  -> detect tool_use
  -> execute tools
  -> append tool_result / attachments
  -> continue next turn or finish
```

如果你能把这条链在脑子里跑起来，这个项目的核心就已经抓住了。

## 5. 为什么 API 层一定要单独读

### 文件

- `src/services/api/claude.ts`
- `src/services/api/client.ts`

### 先看什么

建议重点读：

- streaming 事件组装
- fallback 到 non-streaming
- usage 聚合
- stop reason 与 request id 处理

重点位置：

- `src/services/api/claude.ts:1780-1875`
- `src/services/api/claude.ts:1975-2065`
- `src/services/api/claude.ts:2478-2698`
- `src/services/api/client.ts:88-235`

### 为什么这层不能跳过

很多人会觉得 API 层只是 transport，不值得花时间。

但在这个项目里，API 层直接决定：

- `tool_use` 什么时候被看见
- fallback 后旧流数据怎么丢弃
- usage 怎么累计
- request id 怎么贯穿日志与 resume

所以这层不是“接 SDK”，而是“控制模型协议与本地状态机之间的边界”。

## 6. 工具系统：不要只看 `tools.ts`

### 文件

- `src/Tool.ts`
- `src/tools.ts`
- `src/services/tools/toolExecution.ts`
- `src/services/tools/toolOrchestration.ts`
- `src/services/tools/StreamingToolExecutor.ts`

### 推荐顺序

1. 先看 `Tool.ts`
2. 再看 `tools.ts`
3. 再看 `toolExecution.ts`
4. 再看 `toolOrchestration.ts`
5. 最后看 `StreamingToolExecutor.ts`

### 为什么这样读

因为这 5 个文件分别回答不同问题：

- `Tool.ts`：一个工具最少要被 runtime 了解哪些元信息？
- `tools.ts`：系统里到底有哪些工具，以及 built-in/MCP 怎样合并？
- `toolExecution.ts`：一次工具调用从请求到结果经历了哪些检查？
- `toolOrchestration.ts`：为什么不是所有工具都 `Promise.all`？
- `StreamingToolExecutor.ts`：为什么工具可以在模型还没完全说完时就开始跑？

### 重点位置

- `src/Tool.ts:355-603`
- `src/tools.ts:197-389`
- `src/services/tools/toolExecution.ts:337-520`
- `src/services/tools/toolExecution.ts:635-1175`
- `src/services/tools/toolOrchestration.ts:1-188`
- `src/services/tools/StreamingToolExecutor.ts:35-212`

### 读这层时的核心问题

1. 输入校验和权限判断为什么不在工具自身内部完成？
2. 并发安全是靠工具声明，还是靠调用方猜？
3. streaming fallback 时为什么必须能丢弃一整批正在执行的工具结果？

当你能回答这三个问题时，你就已经不再把这里看成简单 function calling 了。

## 7. MCP：为什么它不是“插件角落”

### 文件

- `src/services/mcp/client.ts`

### 先看什么

优先看：

- MCP tool list 如何变成 `Tool[]`
- permissions 如何复用本地工具逻辑
- URL elicitation retry
- transport 连接管理

重点位置：

- `src/services/mcp/client.ts:1744-1878`

### 应该得到的认识

这个文件最重要的不是“怎么连 MCP server”，而是：

- 远端工具怎样被降维成统一本地抽象
- 一旦降维完成，后续所有机制都能复用

换句话说，这个文件决定项目是“有外部工具接入能力”，还是“本身就是一个工具平台”。

## 8. 如果你要带着问题读，每层该问什么

### 启动层

- 哪些模式只是壳层不同？
- 哪些模式真正改了主逻辑？

### 会话层

- 哪些状态要跨用户消息持久化？
- transcript 何时写、写哪些东西？

### query loop

- 一轮 turn 的递归条件是什么？
- compact 和 budget 是怎么嵌到主循环里的？

### API 层

- 为什么需要自己拼 block？
- fallback 与 usage 聚合怎样保持一致？

### 工具层

- schema / validate / permissions / hooks 的顺序是什么？
- 并发边界是谁来定义？

### MCP 层

- 外部工具如何统一命名？
- 权限、日志、回写如何不分叉？

## 9. 最短阅读路径：只用 90 分钟怎么读

如果你只愿意花 90 分钟，建议这样分配：

1. `src/query.ts` 30 分钟
2. `src/QueryEngine.ts` 20 分钟
3. `src/services/tools/toolExecution.ts` 15 分钟
4. `src/services/api/claude.ts` 15 分钟
5. `src/services/mcp/client.ts` 10 分钟

目标不是吃透每个 if/feature gate，而是建立完整调用图。

## 10. 深读路径：按两轮来

### 第一轮：只抓控制流

只关心：

- 谁调用谁
- 状态在哪一层
- 递归在哪里发生

不要陷进：

- feature gate
- telemetry
- 平台分支
- 内部用户类型分支

### 第二轮：再抓工程质量

再看：

- compact
- budget
- fallback
- interrupt
- transcript consistency
- permission race

这样读，理解成本会低很多。

## 11. 一个实用的源码地图

可以把核心源码压缩成这张地图：

```text
CLI / print shell
  -> QueryEngine.submitMessage()
  -> query()
  -> queryLoop()
  -> services/api/claude.ts streaming
  -> tool_use?
     -> toolExecution / toolOrchestration / StreamingToolExecutor
     -> append tool_result
     -> queryLoop() next iteration
  -> final SDK/result message
```

再往外扩一圈：

```text
StructuredIO / permission prompt / hook callback
MCP client -> dynamic Tool wrapping
Transcript / resume / session storage
Budget / compact / prefetch
```

如果你把这张图吃透，再去读其他目录，速度会快很多。

## 12. 最后一句建议

这个项目不适合“随便点开几个文件感受一下”。

它最值得学的地方恰恰是跨文件的主链路。

所以正确读法不是：

- 看哪个文件名眼熟就进哪个

而是：

- 先沿执行流跑一遍
- 再回头看每一层的局部细节

这样你读到的就不是一堆散乱功能，而是一套真正能落地的 agent runtime 设计。
